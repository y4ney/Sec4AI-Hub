# 核心目标

防范人工智能因遭受恶意操纵，执行未授权命令、滥用工具或提升权限，同时覆盖智能体供应链全流程防护。

# 主动防护：限制人工智能工具调用与执行，落实供应链安全防护

- 制定严格的工具访问控制策略，限定 AI Agent 可执行的工具范围；
- 对提示词 / 脚本仓库、记忆定义实施版本管理与同行评审，参照代码管理标准执行；
- 人工智能调用工具前，需完成函数级身份验证；
- 采用执行沙箱机制，防范人工智能驱动的工具滥用影响生产系统：
  - 将人工智能调用的工具部署在隔离的容器化环境中，禁止其访问敏感系统资源与广域网络；
  - 设置资源限制（CPU、内存），限制系统调用，防范资源滥用或拒绝服务攻击；
  - 每次工具执行完成后，自动重置或销毁沙箱，阻断攻击持久化或横向移动；
  - 若工具通过智能体间协议（如 A2A、MCP）调用，需强制启用沙箱与权限隔离 —— 此类工具调用可能绕过用户界面、授权流程或直接人类监督；
- 对 API 调用与高计算量任务实施限流管控；
- 基于实时风险评分限制人工智能工具执行，若风险因素（如用户行为异常、访问模式可疑）超出预设阈值，立即限制工具使用；
- 对人工智能工具使用实施即时访问（JIT）管控，仅在明确需要时授予访问权限，使用后立即回收；
- 对智能体配置卡、提示词模板、模型 / 工具定义进行数字签名；对智能体及其运行时组件，采用可验证的软件物料清单（AI SBOM/AIBOM/Agent SBOM）。

# 被动响应：监控并防范工具滥用与供应链异常

- 对所有人工智能工具交互行为进行日志记录，确保具备取证追溯能力；
- 检测规避安全策略的命令串联行为；
- 涉及金融、医疗、行政功能的人工智能工具执行操作，需强制获取用户明确批准；
- 留存详细的执行日志，跟踪人工智能工具调用行为，为取证审计与异常检测提供支撑；
- 人工智能生成的高权限代码，执行前需经过人工验证；
- 检测异常的工具执行频率，若发现某一 AI Agent 在极短时间内反复调用同一工具，需标记为攻击可疑行为；
- 监控人工智能工具交互的非预期副作用，检测工具输出是否触发意外的安全敏感操作、行为偏移、数据窃取或能力突发变化。

# 检测监控：防范人工智能资源耗尽与供应链攻击

- 监控智能体工作负载使用情况，实时检测过度的处理请求；
- 对超出预设资源消耗阈值的人工智能进程，强制自动暂停；
- 推行执行控制策略，对人工智能生成代码中绕过预设安全约束的执行尝试进行标记；
- 统计多 AI Agent 的累计资源消耗，防范多个智能体协同消耗过量计算资源，导致系统过载；
- 限制人工智能发起的并发系统修改请求，防范大规模工具执行意外触发拒绝服务（DoS）；
- 通过软件物料清单（AI SBOM/AIBOM/Agent SBOM）监控智能体的供应链依赖，排查安全威胁与入侵指标；
- 开展红队演练，模拟供应链组件被投毒的场景，验证安全防护边界的有效性。
